## 로봇 상태 데이터는 어떻게 전달되는가

많이 사용되는 ROS 기반 시스템에서는 로봇 내부에서 다음과 같은 형태의 메시지가 오갑니다.

![[20260125173441.png]]

ROS에서 로봇은 센서 데이터와 각 디바이스의 상태를 지속적으로 참조하며,
정의된 주기에 따라 메시지를 발행(Publish)합니다.

다만 여기서 중요한 점은 다음과 같습니다.

- **ROS 내부 통신**
  - `.msg / .srv / .action` 정의 기반
  - Binary serialization
- **ROS ↔ 서버 / 클라우드 연동**
  - HTTP, WebSocket, gRPC 등을 사용
  - JSON, Protobuf, Binary 포맷 등 다양한 방식 활용

## 데이터 통신에서 중요한 요소

로봇 관제 시스템에서 핵심은 단순한 데이터 송수신이 아니라,
**인터페이스**, **데이터 포맷**, **Payload 구조**를 어떻게 설계하느냐입니다.

---

# 1. 인터페이스 / 데이터 포맷

로봇 관제 시스템에서 일반적으로 사용되는 인터페이스는 다음과 같습니다.

### REST
- HTTP(S) 기반으로 가장 범용적으로 사용
- Web 기반 시스템과의 연동에 적합
- 일반적으로 JSON 포맷 사용
- 디버깅 및 확장성이 뛰어남

### gRPC
- Protobuf 기반의 고성능 RPC
- 낮은 latency, 높은 처리량
- 명확한 contract 기반 설계에 적합
- 잦은 스키마 변경이나 실험적 개발에는 부담
- 대규모 시스템에서는 클라이언트 관리 비용이 증가

### ROS 통신 구조

ROS는 ROS1과 ROS2에서 통신 구조가 크게 다릅니다.

#### ROS 1 핵심 구성 요소
- **ROS Master**  
  노드 간 연결 정보를 관리하는 중앙 서버
- **Node**  
  센서, 제어, 알고리즘을 수행하는 최소 실행 단위
- **Parameter Server**  
  전역 파라미터 저장소

#### 통신 방식
- **Topic**  
  비동기 1:N 메시지 통신 (센서, 위치 정보)
- **Service**  
  동기 1:1 요청/응답 (짧은 처리)
- **Action**  
  비동기 목표 기반 통신  
  (진행 상태 피드백, 취소 가능, 긴 작업)

#### ROS 1 vs ROS 2
- **ROS 1**
  - XMLRPC (Master)
  - TCPROS / UDPROS
- **ROS 2**
  - DDS 기반
  - Master-less 구조
  - QoS, 실시간성, 안정성 강화

---

### 이벤트 기반 메시지 프로토콜

- RabbitMQ, Kafka 등
- 벤더 독립 이벤트 기반 아키텍처 구성 가능
- Backend 시스템과의 확장성이 뛰어남
- 시스템 규모가 커질수록 강점이 명확해짐

> ROS 2의 DDS는 discovery 과정에서 멀티캐스트를 활용하며,
> 기업 IT 네트워크 환경에서는 제약이 발생하는 경우가 많습니다.  
> 이로 인해 실서비스에서는 Kafka 등 이벤트 기반 메시징을
> 병행하거나 대체 수단으로 사용하는 경우가 많습니다.

다만 네트워크 의존도가 높기 때문에,
장애 대응, 메시지 유실, QoS 설계가 중요합니다.

# 2. Payload 설계

## 1) 실시간 상태 모니터링
- 로봇 온라인 / 오프라인
- 상태 (Idle, Working, Error, Charging 등)
- 배터리 잔량
- 위치 정보
- 속도, 주행 거리
- 미션 진행률

## 2) Map & Localization

이동형 로봇은 대부분 빌딩 내부에서 운용되므로,
지도와 위치 정보는 핵심 모니터링 요소입니다.

- 멀티 사이트 / 멀티 빌딩 맵 관리
- 층, Zone 구분
- 로봇 실시간 위치 추적
- 금지 구역 / 제한 구역 설정
- 임시 장애물 표시

(자세한 내용은 SLAM 섹션에서 별도로 다룹니다.)
