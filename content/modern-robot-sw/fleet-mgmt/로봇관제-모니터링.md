# 1. 인터페이스/데이터 포맷

다음과 같은 인터페이스들이 주로 사용됩니다. gRPC 처럼  데이터 포맷이 자동으로 결정되는 경우들도 존재하지만 대부분이 JSON, XML 기반입니다. 

- REST 
	- 개발에 가장 용이한 포맷으로 HTTP(S) 기반으로 데이터를 주고 받습니다. 
	- web 기반으로 개발이 될 때는 주로 JSON을 사용합니다
- gRPC
	- 속도면에서 강점이 있습니다. 하지만, 로봇 특성상 payload의 변경과 복잡한 시스템을 설계할 수록 다루기 어려워지는 단점들이 있습니다. 스펙이 명확한 프로젝트에 적합합니다
	- 스케일이 커지는 프로젝트에서는 각 클라이언트를 다뤄야 하는 어려움에 직면하기 좋습니다.
- ROS 
	- ROS는 ROS1과 ROS2가 인터페이스/포맷이 다릅니다. 정리해 보면 다음과 같습니다.
	- **핵심 구성 요소 (ROS 1 기준)**
		- **ROS Master**: 노드 간의 통신 연결을 중재하고 관리하는 중심 서버.
		- **Nodes(노드)**: 로봇의 기능(센서, 모터 제어 등)을 수행하는 최소 단위의 프로그램.
		- **Parameter Server**: 네트워크상의 파라미터를 공유하는 저장소. 
	- **2. 통신 방식 (Topic, Service, Action)**
		- **Topic(토픽)**: 퍼블리셔(Publisher)와 서브스크라이버(Subscriber) 간의 비동기식 1:N 메시지 통신 (센서 데이터, 위치 정보).
		- **Service (서비스)**: 클라이언트-서버 간의 동기식 1:1 요청/응답 통신 (짧은 처리).
		- **Action(액션)**: 비동기식 1:1 목표/결과 통신, 중간 피드백 및 취소 가능 (긴 처리). 
	- **3. ROS 1 vs ROS 2 통신 차이**
		- **ROS 1**: XMLRPC(마스터 통신), TCPROS/UDPROS(노드 간 데이터 통신) 사용.
		- **ROS 2**: DDS(Data Distribution Service) 기반의 미들웨어를 사용하여 마스터 없이 노드끼리 직접 통신하며, 더욱 안정적이고 실시간성 확보. 
- 이벤트 기반 메세지 프로토콜
	- Rabbit MQ나 Kafka 혹은 벤더 독립 이벤트 기반 메세지 프로토콜이 있습니다.
	- 강점은 백엔드 서버에서 활용가능한 구조를 만들 수 있습니다.
	- 스케일이 커질수록 메세지 프로토콜이 강점을 가지고 있습니다. 
	- **ROS2의 DDS가 MultiCast 를 강제하기 때문에 실 서비스에서는 잘 쓰기 어렵기 때문에 대안으로 많이 사용됩니다.**
	- 네트워크 의존도가 높기 때문에 

# 2. Payload

아래는 데이터 일반적인 시스템 상에서 주고 받는 payload를 정의해 보았습니다
## 1. 실시간 상태 모니터링

- 로봇 온라인/오프라인 상태
- Status(Idle, Working, Error, Charging...)
- 배터리 용량
- 위치 정보
- 속도, 주행 거리, 미션 진행률

## 2. 맵 & localize

고정된 로봇이 아닌 경우 대부분 빌딩 안에서 이동을 하게 됩니다. 따라서 각 빌딩의 지도가 중요한 모니터링의 요소가 됩니다. 맵 & localize는 SLAM 섹션에서 다시 다루게 될 것인데, 서비스를 위한 지도와 모니터링 요소는 다음과 같습니다.

- 멀티사이트/멀티 빌딩 맵 관리
- 층, zone 구분
- 로봇 실시간 위치 트래킹
- 금지구역, 제한 구역 설정
- 임시 장해물 표시
